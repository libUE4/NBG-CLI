SDK 在 production deployments 中使用 hub-spoke architecture。后台 daemon（hub）负责 session state 和 event routing，spoke workers 执行 agent loop，clients（CLI、VS Code、JetBrains 等）通过 WebSocket 作为 peers 接入。

## 为什么使用 Hub-Spoke？

Single-process architecture 适合简单 scripts，但遇到以下需求时会变得脆弱：

- 窗口关闭或 CLI 退出后仍然保留 sessions
- 多个 clients 查看并控制同一个 session
- 没有 client 连接时仍能运行 scheduled agents
- 通过 process isolation 避免失控 agent 卡住 UI

Hub-spoke model 通过把 coordination、execution 和 client UI 分离来解决这些问题。

## 三个角色

<CardGroup cols={3}>
  <Card title="Hub" icon="circle-nodes">
    每台机器一个 singleton daemon。负责协调 sessions、路由 events 和 approvals、管理 schedules，并在 clients 之间 broker capabilities。Hub 不运行 agent loop。
  </Card>
  <Card title="Spoke" icon="gear">
    运行 `@cline/core` 的 worker process。负责执行 agent loop、调用 tools、流式输出，并把 events 回报给 hub。Spoke 由 daemon 拥有，不归属于某个 client。
  </Card>
  <Card title="Client" icon="display">
    CLI、VS Code、JetBrains 或任意 custom app。负责发现 hub、通过 WebSocket 注册、attach 到 sessions、发送用户输入，并接收 streamed events。
  </Card>
</CardGroup>

保持架构清晰的规则是：clients 参与，spokes 执行，hub 协调。不同角色不应互相重叠。

## 通信流程

Clients 通过 WebSocket 连接 hub。Hub 路由 commands 并流式转发 events。Spokes 只向 hub 回报，不直接连接 clients。

<Steps>
  <Step title="Client 发现或启动 hub">
    Hub 是每台机器上的 singleton daemon。如果当前没有运行中的 hub，`ClineCore` 会自动启动。Discovery 使用 `~/.cline/locks/hub/owners/` 下的 lock files。
  </Step>
  <Step title="Client 注册并 attach 到 session">
    Client 通过 WebSocket 发送 `client.register` command，声明自己的 capabilities，例如 shell access、file editing、diff viewing 等。随后它会创建或 attach 到一个 session。
  </Step>
  <Step title="Hub 分配 spoke 执行">
    Hub 会 spawn 或分配一个 spoke worker 来运行 agent loop。Spoke 执行 tools、流式输出部分结果，并处理 abort/cancel。
  </Step>
  <Step title="Hub 将 events 分发给所有 attached clients">
    当 spoke 产生 events，例如 text deltas、tool calls 和 usage 时，hub 会把它们转发给 attach 到该 session 的所有 clients。Clients 可以随时加入或离开，不会中断执行。
  </Step>
  <Step title="Client 断开后 session 继续运行">
    如果 CLI 退出或窗口关闭，spoke 会继续执行。另一个 client 可以 attach 到同一 session，并接上正在进行的 event stream。
  </Step>
</Steps>

## Backend Modes

`ClineCore` 会根据配置选择 execution mode：

```typescript
const cline = await ClineCore.create({
  clientName: "my-app",
  backendMode: "auto",
})
```

| Mode | 行为 |
|------|----------|
| `auto` | 优先使用可用的兼容 local hub；不可用时回退到 local in-process execution。默认模式。 |
| `hub` | 要求可访问的兼容 WebSocket hub；如果无法连接会抛错。 |
| `remote` | 要求显式 remote WebSocket hub endpoint，适用于 hub 不在用户机器上的 deployments。 |
| `local` | 始终使用 local in-process execution 和 local SQLite/file storage。没有 hub，也没有 shared sessions。 |

### When to Use Each Mode

Use `local` for:
- Simple scripts and one-off tasks
- Testing and development
- Environments where a background process isn't appropriate

Use `hub` (or `auto`) for:
- Session persistence across client restarts
- Multiple clients sharing the same session
- Scheduled agents
- Connector integrations (Telegram, Slack, etc.)

Use `remote` for:
- Cloud deployments where the hub runs on a server
- Team-shared hub instances

## Capability Brokerage

When multiple clients are attached to a session, the hub routes capability requests to whichever client can handle them. Clients advertise their capabilities at registration.

For example, VS Code might register `open-file`, `reveal-diff`, and `run-build` capabilities. The CLI might register `shell` and `run-tests`. When the agent needs to open a diff, the hub routes the request to VS Code. When it needs to run a shell command, the hub routes to the CLI.

This means a session gets richer as more clients join. A CLI session can later be enriched by VS Code attaching -- an IDE selection can be sent into a running terminal session, or a diff created by one client can be opened by another.

## Session Persistence

Sessions are stored with full conversation history, tool call records, and metadata. The hub maintains a SQLite index for efficient listing and JSON snapshots as the source of truth for each session's state.

```
~/.cline/data/sessions/
  sessions.db                    # SQLite index
  [session-id].json              # Authoritative session record
```

Sessions can have multiple participants, each tracked with a role (`creator`, `participant`, `observer`). The session persists independently of any single client's lifecycle.

## Managing the Hub

当 `ClineCore` 需要 hub 时会自动启动。也可以手动管理：

```bash
nbg hub start      # Start the hub daemon
nbg hub stop       # Stop it
nbg hub status     # Check if running
nbg hub ensure     # Start if not running, return URL
```

Hub 默认监听 `127.0.0.1:25463`，日志写入兼容路径 `~/.cline/logs/hub-daemon.log`。

## Multi-Client Access

Multiple clients can connect to the same hub and attach to the same session simultaneously:

```bash
# Terminal 1: start a task
nbg "Work on feature X"

# Terminal 2: attach to the same session from VS Code or another CLI

# Terminal 3: Telegram connector
nbg connect telegram -k $TOKEN

# All three share the same hub and can access the same sessions
```

## Process Isolation

If a client crashes, the hub and its running spokes are unaffected. The session continues executing in the spoke worker. When the client reconnects (or a different client attaches), it receives the full session history and resumes the live event stream.

The spoke also has a process boundary from the hub itself. A runaway model call in a spoke doesn't degrade the hub's ability to coordinate other sessions, route approvals, or fan out events.
