NBG 支持 VS Code multi-root workspaces，可以在同一个窗口中管理多个项目文件夹或仓库。无论是 monorepo 还是多个独立 Git 仓库，NBG 都可以跨 workspace 读取文件、修改代码并运行命令。

<Warning>
Multi-root workspaces 当前有两个限制：
- **NBG 规则** 只在 primary workspace folder 中生效
- **[Checkpoints](/core-workflows/checkpoints)** 会被禁用，回到单文件夹 workspace 后恢复

详情见 [当前限制](#当前限制)。
</Warning>

## 理解 Multi-root workspaces

开始前，先区分两种常见的关联项目组织方式。

### 为什么使用 multi-root workspace？

NBG 可以完成跨多个项目或仓库的任务：

- **重构**：更新 API contract，并修复多个仓库中的所有 consumers
- **功能开发**：实现同时涉及 frontend、backend 和 shared code 的功能
- **依赖更新**：协调多个相关项目的版本升级
- **文档生成**：生成引用多个仓库代码的文档

**示例 prompt：**
```text
更新 contracts repo 中的 User 类型，然后同步更新 frontend 和 backend，
让它们使用新字段。确保 API 会校验新的必填字段。
```
## 设置 Multi-root workspace

### Monorepos 与多个仓库

**Monorepo**：一个 Git 仓库包含多个项目或 packages，所有代码共享同一套版本历史。

<Files>
  <Folder name="my-company" defaultOpen>
    <Folder name=".git" />
    <Folder name="packages" defaultOpen>
      <Folder name="web" icon="react">
        <File name="..." />
      </Folder>
      <Folder name="api" icon="node-js">
        <File name="..." />
      </Folder>
      <Folder name="shared">
        <File name="..." />
      </Folder>
    </Folder>
    <File name="package.json" />
  </Folder>
</Files>

**多个仓库**：多个独立 Git 仓库各自拥有版本历史，并一起打开在同一个 VS Code workspace 中。

```text
~/projects/
├── fullstack.code-workspace   # Workspace config file
├── frontend/                  # git@github.com:acme/frontend.git
│   └── .git/
├── backend/                   # git@github.com:acme/backend.git
│   └── .git/
└── contracts/                 # git@github.com:acme/api-contracts.git
    └── .git/
```

NBG 支持这两种模式，也支持部分文件夹是 Git 仓库、部分不是的 hybrid setup。关键区别是：多个仓库模式下，每个文件夹都有自己的 `.git` 目录，NBG 会独立跟踪它们。

### 添加文件夹到 workspace

可以通过几种方式添加文件夹：

- **File menu**：在 VS Code 中使用 `File > Add Folder to Workspace`
- **拖拽**：把文件夹直接拖到 VS Code file explorer
- **Workspace file**：创建 `.code-workspace` 文件，推荐团队使用
- **Command palette**：运行 `Workspaces: Add Folder to Workspace`

详细步骤见 [Microsoft multi-root workspace guide](https://code.visualstudio.com/docs/editor/multi-root-workspaces)。

## 使用多个仓库

当你在一个 workspace 中打开多个独立 Git 仓库时，NBG 会把每个仓库视为拥有独立版本控制的项目。

### NBG 会跟踪什么

对每个 workspace folder，NBG 会检测：

| 属性 | 说明 |
|----------|-------------|
| **Path** | 文件夹绝对路径 |
| **Name** | 从文件夹名或 workspace file 推导 |
| **VCS Type** | Git、Mercurial 或 None |
| **Commit Hash** | 当前 HEAD commit（Git/Mercurial repos） |

这意味着 NBG 可以理解 frontend 和 backend 可能处于不同 commit、不同 branch，甚至使用不同版本控制系统。

<Note>
虽然 NBG 会检测所有 workspace folders 的 VCS 信息，但部分功能只使用 **primary workspace**（第一个 folder）：[规则](/customization/cline-rules)、[skills](/customization/skills#triggering-skills-with-slash-commands)，以及 `@git` mentions 等 [Git 相关功能](/core-workflows/working-with-files)。
</Note>

## 跨 workspace 引用文件

### 自然语言引用

NBG 可以理解对 workspace 的自然语言引用：

```text
"Read the package.json in the frontend folder"
```

```text
"Compare the user model in backend with the TypeScript types in contracts"
```

```text
"Search for TODO comments across all workspaces"
```

### Workspace hints 语法

需要显式引用时，使用 `@workspace:path` 语法：

| 语法 | 说明 |
|--------|-------------|
| `@frontend:src/App.tsx` | "frontend" workspace 中的文件 |
| `@backend:server.ts` | "backend" workspace 中的文件 |
| `@contracts:types/` | "contracts" workspace 中的文件夹 |

该语法特别适合：
- 多个 workspaces 中有同名文件
- 需要明确指定目标项目
- NBG 需要消除歧义

### Workspace 名称来源

Workspace 名称来自：
1. `.code-workspace` 文件中的 `name` 字段（如果指定）
2. 文件夹名称（默认）

如果两个文件夹同名，可以追加数字，或在 workspace file 中指定唯一名称。

## 常见配置

### Monorepo 开发

```text
~/projects/my-app/
├── my-app.code-workspace      # Workspace config file
├── web/          (React frontend)
├── api/          (Node.js backend)  
├── mobile/       (React Native)
└── shared/       (Common utilities)
```

所有文件夹共享同一套 Git 历史，跨 package 的变更可以作为一个整体提交。

**示例 prompt：** *"更新 web 和 mobile apps 中的 API endpoint，使其匹配新的 backend route"*

### 独立仓库的微服务

```text
~/projects/services/
├── services.code-workspace    # Workspace config file
├── user-service/       (git: github.com/acme/user-service)
├── payment-service/    (git: github.com/acme/payment-service)
├── gateway/            (git: github.com/acme/api-gateway)
└── proto/              (git: github.com/acme/service-protos)
```

每个 service 都有自己的仓库。NBG 可以更新 proto definitions，并跨所有 services 重新生成 clients。

**示例 prompt：** *"在 proto 的 UserProfile message 中添加新字段，然后更新 user-service 和 gateway 处理该字段"*

### 带 shared contracts 的 full-stack 项目

```text
~/projects/fullstack/
├── fullstack.code-workspace   # Workspace config file
├── client/         (git: github.com/acme/web-client)
├── server/         (git: github.com/acme/api-server)
└── types/          (git: github.com/acme/shared-types)
```

types 仓库定义 client 和 server 共同使用的 interfaces。更新 type 时，NBG 可以同时修复两端 consumers。

### Hybrid setup

```text
~/projects/project/
├── project.code-workspace     # Workspace config file
├── main-app/       (git: github.com/acme/main-app)
├── vendor/         (no VCS - vendored dependencies)
└── scripts/        (no VCS - local automation)
```

该模式混合了仓库和普通文件夹。NBG 会按每个 folder 的配置适配。

## 当前限制

Multi-root workspace mode 下有两个功能限制：

### NBG 规则

[规则](/customization/cline-rules)（`.clinerules/` directory）只在 **primary workspace**（workspace 中第一个 folder）中生效。其他 workspace folders 中的 rules 会被忽略。

**替代方案：** 把 shared rules 放在 primary workspace，或使用全局 rules（`~/Documents/Cline/Rules/`），让它们对所有 workspace 生效。

### Checkpoints

[Checkpoints](/core-workflows/checkpoints) 在 multi-root workspace mode 下会被禁用。发生时 NBG 会显示警告。

**原因：** Checkpoints 使用 shadow Git repository 跟踪变更。多个仓库拥有独立 Git 历史，跨仓库协调 checkpoints 的复杂度目前还未支持。

**替代方案：** 使用常规 Git workflow。频繁提交，或为实验性工作创建分支。

回到单文件夹 workspace 后，这两个限制会恢复正常。

## 最佳实践

### 组织 workspace

1. **把相关项目放在一起**，尤其是经常需要协同修改的项目
2. **使用 workspace file**，让团队成员复现同一套配置
3. **清晰命名 folders**，让 workspace hints 更直观
4. **提前考虑 primary workspace**，用于放置 NBG rules

### 高效提示

- **在关键处具体说明**：*"更新 backend workspace 中的 user model"*
- **描述项目关系**：*"frontend 使用 contracts workspace 中的 types"*
- **说明跨 workspace 变更**：*"这需要同时更新 web 和 mobile"*
- **为大型代码库限定搜索范围**：*"只在 frontend workspace 中搜索 'TODO'"*

### 处理大型 workspace

- 尽量把大任务拆成针对具体 workspace 的操作
- 使用 [Plan mode](/core-workflows/plan-and-act) 让 NBG 先理解结构
- 添加 `.clineignore` 文件减少噪声、加快扫描，并让 NBG 聚焦源码：

```text
# 依赖
**/node_modules/

# 构建输出
**/dist/
**/build/

# VCS 元数据
**/.git/
```

更多模式和注意事项见 [.clineignore 文件指南](/customization/clineignore)。
