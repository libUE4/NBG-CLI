当会话接近模型上下文窗口限制时，NBG 会自动总结已有上下文，释放空间并继续执行任务。

## 工作方式

NBG 会在会话中监控 token 使用量。接近限制时，它会：

1. 为已发生的工作生成完整摘要
2. 保留技术细节、代码修改和关键决策
3. 用摘要替换较长的历史消息
4. 从中断点继续执行

发生自动压缩时，你会看到一次 summarization tool call。它和其他 API 调用一样会产生费用记录。

## 为什么重要

没有摘要时，达到上下文限制通常只能截断旧消息，容易丢失关键上下文。

启用自动压缩后：

- 技术决策和代码模式会被保留
- 文件修改和项目上下文保持连续
- NBG 可以记住已经完成的工作
- 长任务和大项目不容易被上下文窗口打断

<Tip>
自动压缩特别适合长任务。结构化任务列表能让摘要后的进度更稳定，帮助 NBG 跨多个上下文窗口保持方向一致。
</Tip>

## 成本说明

摘要会利用会话中已有的 prompt cache，成本通常接近一次普通 tool call。

由于多数 input tokens 已被缓存，主要成本来自摘要生成的 output tokens。

<Note>
部分模型不支持 agentic summarization 时，即使启用自动压缩，NBG 也会回退到标准规则式上下文截断。
</Note>

## 恢复上下文

可以使用 [checkpoints](/core-workflows/checkpoints) 恢复到摘要发生前的任务状态。只要 checkpoint 可用，就可以回滚到更早的上下文。

编辑 summarization tool call 之前的消息也类似，会把会话恢复到对应位置。
