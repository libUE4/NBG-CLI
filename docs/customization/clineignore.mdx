`.clineignore` 文件用于告诉 NBG 在分析代码库时跳过哪些文件和目录。它的工作方式类似 `.gitignore`：在项目根目录创建名为 `.clineignore` 的文件，加入需要排除的模式，NBG 就会忽略它们。

## 为什么重要

没有 `.clineignore` 时，NBG 可能把整个项目载入上下文，包括依赖、构建产物和生成文件。这会浪费 tokens、增加成本，并把真正有用的上下文挤出窗口。

添加 `.clineignore` 可以把起始上下文从 200k+ tokens 降到 50k 以下。这意味着响应更快、成本更低，也更容易有效使用较小、较便宜的模型。

## 创建 .clineignore

在项目根目录创建名为 `.clineignore` 的文件：

```text
# Dependencies
node_modules/
**/node_modules/

# Build outputs
/build/
/dist/
/.next/
/out/

# Testing artifacts
/coverage/

# Environment variables
.env
.env.*

# Large data files
*.csv
*.xlsx
*.sqlite

# Generated/minified code
*.min.js
*.map
```

## 模式语法

`.clineignore` 使用与 `.gitignore` 相同的 pattern syntax：

| Pattern | 匹配内容 |
|---------|---------|
| `node_modules/` | `node_modules` 目录 |
| `**/node_modules/` | 任意层级的 `node_modules` |
| `*.csv` | 所有 CSV 文件 |
| `/build/` | 仅项目根目录下的 `build` 目录 |
| `*.env.*` | 类似 `.env.local`、`.env.production` 的文件 |
| `!important.csv` | 例外：不要忽略这个文件 |

以 `#` 开头的行是注释，空行会被忽略。

## 应该排除什么

可以从以下类别开始，再按项目情况调整：

**几乎总是应该排除：**

- 包管理器目录（`node_modules/`、`vendor/`、`.venv/`）
- 构建产物（`dist/`、`build/`、`.next/`、`out/`）
- 覆盖率报告（`coverage/`）
- 体积较大的 lock files（`package-lock.json`、`yarn.lock`）

**存在时建议排除：**

- 大型数据文件（`.csv`、`.xlsx`、`.sqlite`、`.parquet`）
- 二进制资源（图片、字体、视频）
- 生成代码（API clients、protobuf outputs、minified bundles）
- 包含密钥的环境文件（`.env`、`.env.local`）

**保持可访问：**

- 你正在主动修改的源代码
- NBG 需要理解的配置文件（`tsconfig.json`、`package.json`）
- 文档和 README
- 测试文件（NBG 通常需要它们作为上下文）

## 工作方式

当 NBG 扫描项目以构建上下文时，会用 `.clineignore` patterns 检查每个文件路径。匹配的文件会从以下范围中排除：

- NBG 启动任务时看到的文件列表
- 对话中的自动上下文收集
- NBG 搜索相关代码时的搜索结果

你仍然可以通过 [@ mentions](/core-workflows/working-with-files) 显式引用被忽略的文件。如果输入 `@/node_modules/some-package/index.js`，即使 `node_modules/` 已在 `.clineignore` 中，NBG 也会读取这个指定文件。ignore rules 控制的是自动加载，不是显式访问。

<Note>
`.clineignore` 独立于 `.gitignore`。即使某些文件被 Git 跟踪，只要与 NBG 上下文无关，例如大型测试 fixtures 或数据文件，也应该加入 `.clineignore`。
</Note>

## 建议

- 尽早在项目中添加 `.clineignore`。先使用较宽的排除规则，再逐步收窄，通常比事后排查上下文膨胀更容易。
- 添加 `.clineignore` 后，查看 task header 中的 token usage，差异通常很明显。
- 如果 NBG 似乎缺少某个文件的上下文，检查它是否被 ignore patterns 排除了。
- 对于 monorepos 或 multi-root workspaces，每个 workspace root 都可以有自己的 `.clineignore`。详情见 [Multi-Root Workspaces](/features/multiroot-workspace)。
