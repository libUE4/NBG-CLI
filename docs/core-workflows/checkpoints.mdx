Checkpoints 让你可以在不丢失对话的情况下撤销代码变更。每次 NBG 修改文件或运行命令时，都会保存一份项目文件快照。你可以恢复到任意 checkpoint，同时保留已经积累的对话上下文。

这会改变你使用 NBG 的方式。你不必在批准前逐项检查所有改动，可以让 NBG 更快推进，并在结果不理想时回滚。试错成本会显著降低。

<Tip>
Checkpoints 默认启用。如需关闭，见 [启用或禁用 Checkpoints](#enable-or-disable-checkpoints)。
</Tip>

## 工作方式

NBG 会维护一个独立于项目真实 Git 历史的 shadow Git repository。每次工具调用后，例如文件编辑或命令执行，NBG 都会把当前文件状态提交到这个 shadow repo。你的主 Git repository 不会被改动。

这意味着：

- Git 历史保持干净，并继续由你控制
- Checkpoints 会捕获所有文件，包括未被 Git 跟踪的文件
- 可以恢复到任务中的任意时刻，不影响你已经创建的 commits
- Checkpoints 会跨编辑器会话保留

每个 checkpoint 都会记录当时完整的文件状态。如果 NBG 连续编辑三个文件，你会得到三个 checkpoints，并且可以独立恢复到其中任意一个。

## 启用或禁用 Checkpoints

Checkpoints 默认启用。切换方式如下：

1. 打开 NBG settings（NBG sidebar 中的齿轮图标）
2. 滚动到 **Feature Settings** 区域
3. 切换 **Enable Checkpoints**

<Note>
对于非常大的 repository，checkpoints 可能占用较多存储，并因为每次工具调用后都要提交文件快照而拖慢 NBG。如果观察到性能问题，可以考虑关闭。
</Note>

## 查看和比较变更

每次工具调用后，对话中都会出现一个 checkpoint indicator。你可以寻找标记为 **Checkpoint** 的 bookmark icon，以及连接到 **Compare** 和 **Restore** 按钮的虚线。

点击 **Compare** 会打开 diff view，展示该 checkpoint 处的具体变更。它会使用编辑器的 diff viewer，帮助你查看所有受影响文件中的新增、删除和修改。

当你想先理解 NBG 的改动，再决定是否保留时，这很有用。你可以先 review diff，然后继续或恢复。

## 恢复 Checkpoints

点击任意步骤旁边的 **Restore** 可以打开恢复菜单。你有三个选项：

| 选项 | 作用 | 适用场景 |
|--------|--------------|----------------|
| **Restore Files** | 把项目文件恢复到该 checkpoint 的快照 | 撤销代码变更，同时保留对话 |
| **Restore Task Only** | 删除该点之后的消息，不影响文件 | 保留当前代码，换一种 prompt 继续 |
| **Restore Files & Task** | 恢复文件，并删除该点之后的消息 | 从一个确定的好状态重新开始 |

具体选择取决于问题类型：

- 如果对话方向正确，但代码变更破坏了某些行为，使用 **Restore Files**。NBG 会保留已经讨论过的上下文，并尝试不同实现。

- 如果代码变更本身不错，但对话方向偏了，使用 **Restore Task Only**。你会保留文件，并用新的方式引导后续对话。

- 如果想从干净状态重新开始，使用 **Restore Files & Task**。它会把文件和对话都重置到该 checkpoint。

## 何时使用 Checkpoints

| 场景 | 建议操作 |
|----------|-------------------|
| NBG 重构代码后破坏了行为 | Restore Files，然后要求换一种方案 |
| 尝试多个解决方案 | Compare 每个 checkpoint，恢复到最佳方案 |
| NBG 误解了你的意图 | Restore Files & Task，然后重新描述需求 |
| 想尝试不同 prompt | Restore Task Only，保留文件后重新提交 |
| 在提交 Git 前 review 变更 | 使用 Compare 检查，然后手动 commit |
| 测试高风险变更 | 让 NBG 继续推进，失败后恢复 |

## 配合 Auto-Approve 使用

Checkpoints 让 [auto-approve](/features/auto-approve) 更实用。没有 checkpoints 时，auto-approve 风险较高，因为你可能在注意到问题前已经积累很多变更。有了 checkpoints，就可以让 NBG 自主推进，并在需要时回滚。

典型流程：

1. 为文件编辑和命令启用 auto-approve
2. 让 NBG 快速完成任务
3. Review 最终结果
4. 如果有问题，恢复到最后一个可用 checkpoint
5. 给 NBG 更具体的指导

这种方式比逐项 review 每个变更更快，而 checkpoints 提供了回滚保障。

## Checkpoints 与消息编辑

消息编辑功能会与 checkpoints 集成。当你编辑之前的消息并选择 **Restore All** 时，NBG 会先把文件恢复到该消息对应的 checkpoint，再重新提交编辑后的消息。

这可以让你一次性修正表达不清的 prompt，并撤销由它导致的所有变更。
