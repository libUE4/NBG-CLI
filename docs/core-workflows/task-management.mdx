你与 NBG 的每次交互都发生在 task 中。Tasks 是独立 work sessions，会记录完整对话、代码变更、命令执行和决策。

## 什么是 Tasks？

当你向 NBG 提交 prompt 时，一个 task 就开始了。Prompt 定义目标，NBG 会通过对话、代码变更和 tool use 朝目标推进。初始 prompt 的质量会直接影响 NBG 的表现；清晰、具体的 prompts 通常带来更好的结果。

每个 task 都会：

- 从你的 prompt 开始，并在对话中逐步构建上下文
- 拥有唯一 identifier 和专用 storage directory
- 保存完整 conversation history
- 跟踪 token usage、API costs 和 execution time
- 可以跨 sessions 中断和恢复
- 通过 Git-based snapshots 为文件变更创建 [checkpoints](/core-workflows/checkpoints)

<Tip>
想让 NBG 给出更好的结果？从清晰、范围明确的 tasks 开始，并把无关工作放到独立对话中。
</Tip>

## 设定 Task 范围

每个 task 都带有自己的上下文：conversation history、已经做出的 decisions，以及 session 中逐步建立的理解。Task 范围会直接影响 NBG 的帮助质量。

可以按这个原则理解：**一个 task = 一个目标**。`Implement user authentication` 是一个 task。`Fix an unrelated CSS bug` 是另一个 task，即使你是在处理 auth 时发现了这个 CSS bug。

聚焦的 task 通常效果更好。当一个 task 试图覆盖太多无关目标时，上下文会变得混乱，响应相关性也会下降。

<Note>
如果不确定，优先新建 task。你始终可以在 task history 中找到之前的 sessions。
</Note>

### Context Window

每个 AI model 都有 context window，也就是一次能处理的信息上限。可以把它理解为 NBG 当前 task 的 working memory。

随着工作推进，context window 会逐渐填入：

- 你的 prompts 和 NBG 的 responses
- NBG 读取或编辑的文件内容
- Command outputs 和 tool results
- 指导 NBG 行为的 system instructions，包括 [NBG Rules](/customization/cline-rules)

当 context window 接近上限时，NBG 会自动压缩较早的对话内容来腾出空间。这意味着很长的 tasks 可能会丢失一些早期细节，但 NBG 会保留最重要的上下文。

这就是 task scoping 重要的原因：聚焦 task 会把相关信息留在 context window 中；范围过大的 task 会用噪声填满窗口，把有用上下文挤出去。

如果简单 prompts 的起始上下文也很高，可以添加 [`.clineignore`](/customization/clineignore) 文件，排除依赖、构建产物和其他 NBG 不需要的文件。这个兼容文件能显著降低 baseline token usage。

<Tip>
对于 long-running tasks，可以启用 [Auto-Compact](/features/auto-compact)，让 NBG 在工作过程中智能管理上下文。
</Tip>

### New Task vs. Continue

一开始可能不容易判断何时新建 task、何时继续。随着使用 NBG 增多，你会逐渐形成判断。可以先参考这张表：

| 场景 | 动作 | 原因 |
|----------|--------|-----|
| 切换到不同 feature | **New task** | 上下文干净，响应更聚焦 |
| 基于 NBG 刚完成的工作继续 | **Continue** | 保留 shared understanding |
| NBG 持续偏离方向 | **New task** | 继续对抗混乱上下文会浪费时间 |
| 围绕同一批文件迭代 | **Continue** | Conversation history 有帮助 |
| 反复解释哪些内容要忽略 | **New task** | 混乱上下文会降低质量 |
| 细化 NBG 上一次输出 | **Continue** | 保留 momentum 和 decisions |

要开始新 task，可以点击 NBG sidebar 中的 **+** 按钮，或使用 `/newtask` command。文件变更会通过 [checkpoints](/core-workflows/checkpoints) 保留，你也可以随时从 history 引用之前的 tasks。

## 理解 Task 成本

所有 cloud-based AI models 都按 tokens 收费，tokens 是模型处理文本的计量单位。NBG 会自动跟踪这些成本，并显示在 task header 中，方便你在工作时监控花费。

### How Costs Are Calculated

与 NBG 交互时，模型会处理：

- **Input tokens**：你的 prompts、文件内容、conversation history 和 system instructions
- **Output tokens**：模型 responses、code suggestions 和 tool calls

Cloud providers 通常按每百万 tokens 收费，output tokens 往往比 input 更贵。一些 providers 还支持 **prompt caching**，当相同上下文（如 NBG rules 或大文件）出现在多个请求中时可以降低成本。可用时，NBG 会自动跟踪 cache savings。

Task header 中显示的 estimated cost 会在每次 API request 后更新。该估算使用所选 provider 的 pricing 信息，最终账单可能因 provider 的取整或计费方式略有差异。

### When You Pay

使用 Anthropic、OpenAI、OpenRouter 或 Google 等 cloud providers 时，需要为 AI usage 付费。成本差异可能很大：

| Provider Type | Billing Model |
|--------------|---------------|
| **NBG Gateway / managed provider** | Pay-per-use or organization-managed billing, depending on deployment |
| **Direct API keys** | Billed by your provider (Anthropic, OpenAI, etc.) |
| **OpenRouter/Requesty** | Aggregated billing across multiple models |
| **Local models** | Free (you provide the hardware) |

如果使用自己的 API keys，请查看 provider pricing page 获取当前价格。价格会频繁变化，并且因 model 而异。

### Free Options

暂时不想付费？NBG 支持几种低成本路径：

- **Free models**：使用兼容 managed provider 时，在 model selector 中搜索 `free`。可用模型会显示 **FREE** 标签，适合学习和实验。
- **Free tiers**：一些 providers 在使用自有 API key 时提供有限免费额度。
- **Local models**：在自己的硬件上运行模型，没有按请求计费。

### Self-Hosted Models

本地运行模型意味着没有 API 成本，唯一成本是运行模型所需的硬件。

要有效运行 local models，通常需要：

- **32GB RAM minimum** for entry-level models (4-bit quantization)
- **64GB RAM** for better quality (8-bit quantization)
- **128GB+ RAM** for cloud-competitive performance

代价是速度。Local models 在常见硬件上通常每秒 5-20 tokens，而 cloud APIs 往往能达到每秒数百 tokens。Local models 也需要更多 setup 和 configuration。

<Tip>
如果你有合适硬件，local models 可以提供完全隐私下的无限实验。入门见 [Running Models Locally](/running-models-locally/overview)。
</Tip>

对大多数用户来说，先从 free cloud models 开始，再按需转向付费选项，通常能在成本、速度和能力之间取得更好平衡。如何选择适合工作流的模型，见 [认证与模型选择](/getting-started/authorizing-with-cline)。


## Task History

每个 task 都会自动保存到本机。你可以重新查看过去的 conversations、恢复中断的工作，或引用之前 sessions 中成功的方法。

### Finding Your History

点击 NBG sidebar 中的 **History** 按钮（右上角 clock icon）打开 history view。你会看到所有历史 tasks，包括 initial prompt、timestamp 和 token usage。展开 task card 可以预览 conversation。

### Searching Tasks

使用 history view 顶部的 search bar 查找特定 tasks。Fuzzy search 会搜索所有内容：你的 prompts、NBG responses、code snippets 和 file names。

可以按以下方式排序：

- **Newest/Oldest**：按时间浏览
- **Most Expensive/Most Tokens**：查找资源消耗高的 tasks
- **Most Relevant**：搜索特定内容时使用
- **Favorites**：只显示 starred tasks

<Tip>
可以策略性使用 favorites。将代表成功模式、好 prompts 或以后可能引用的复杂工作的 tasks 标星。Favorited tasks 会受到保护，不会被删除。
</Tip>

## Resuming Tasks

NBG 可以带着完整上下文恢复中断的 tasks：

1. 从 history 打开 task
2. NBG 加载完整 conversation
3. 根据 [checkpoints](/core-workflows/checkpoints) 检查 file states
4. Task 会在理解中断位置的情况下继续
5. 如有需要，提供额外上下文

这可以跨 sessions 工作。即使关闭编辑器并在几天后回来，NBG 也能从离开的地方继续。
